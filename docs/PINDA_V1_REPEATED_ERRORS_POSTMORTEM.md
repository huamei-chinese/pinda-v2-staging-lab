# Pinda V1 反复错误与低效率原因复盘

## 1. 最近反复错误列表

### Pinda V1 正式项目

1. 修复没有覆盖真实页面数据源。
2. 测试数据修好了，但真实页面仍然读取旧数据源。
3. 没有一开始锁定唯一渲染函数。
4. 旧字段和新字段混用：`word` / `phrase` / `sentence` / `stage` / `type` / `pos` / `segment_type`。
5. 学习模块、素材类型、句子内部词块类型、词性标签混在一起。
6. 测试覆盖了小包，但没有覆盖真实旧数据。
7. 文档标准更新快于前端真实实现。
8. 本地测试修改和正式可提交修改没有分层。
9. 人工预览发现的问题比自动测试更真实。

### 素材工厂

1. 没有先离线验证，就直接批量 TTS。
2. 编码问题反复出现，越南语和拼音声调容易乱码。
3. TTS 错误处理不完整，HTTP 200 不等于成功。
4. 自检规则制造误判，例如安全关键词被自检报告自身命中。
5. 音频质量不能只看 MP3 是否存在，短词 slow 需要人工听审或 ASR 抽查。
6. 需求变化后，旧生成策略没有及时收敛。
7. 临时 PowerShell / 内联 Python 太多，复用性差。

## 2. Pinda V1 低效率原因

Pinda V1 的低效核心原因是：真实数据源、渲染出口、素材标准、测试路径没有统一。

具体表现：

1. 页面实际读取 `public/lesson-high-frequency-topics.js`，但测试优先覆盖了本地 V2 小包。
2. 句子拆解标签没有一开始收敛到唯一函数 `displaySegmentLabel(segment)`。
3. 旧 `phrase` 字段既表示学习模块，又表示词块，又曾被映射为“生词”。
4. 人工截图问题没有第一时间沉淀为回归测试。
5. 本地预览改动、正式代码修复、素材标准文档在同一工作区混杂。

## 3. 素材工厂低效率原因

素材工厂的低效核心原因是：过早进入昂贵 TTS 生成，把规则、编码、自检、内容质量放到了后面。

正确顺序应该是：

1. 离线 manifest 检查。
2. 编码和字段检查。
3. 1 条 TTS probe。
4. 少量样本人工听审。
5. 批量生成。
6. 独立自检。

## 4. 真实根因分析

1. 标准先变了，但真实加载路径没有同步收敛。
2. 测试验证了“新理想路径”，没有验证“用户实际看到的路径”。
3. 多个函数各自判断同一个概念，造成旧 fallback 反复漏出。
4. 对素材包的验收偏重文件存在，偏轻语义、音频质量和前端渲染效果。
5. GitHub 固化前缺少统一的本地总闸门。

## 5. 以后禁止重复发生的错误

1. 禁止未定位真实数据源就修改 UI。
2. 禁止只修测试数据、不查真实页面数据。
3. 禁止 sentence segment 标签直接显示 `pos`。
4. 禁止句子拆解卡片显示“生词”“关键词”或单独的“短语”。
5. 禁止未本地预览通过就 commit / push / PR。
6. 禁止素材工厂跳过离线 manifest 检查直接批量 TTS。
7. 禁止把 HTTP 200 当作 TTS 成功。
8. 禁止把自检报告自身命中的安全词误判为素材泄密。

## 6. 防错措施

| 错误 | 防错措施 |
| --- | --- |
| 修错数据源 | 每次先回答“页面文字来自哪个文件、字段、函数” |
| 标签 fallback 泄漏 | 建立唯一函数 `displaySegmentLabel(segment)` |
| 小包测试通过但真实页面失败 | 测试必须覆盖真实旧数据源 |
| 截图问题反复 | 每个截图问题必须变成回归测试 |
| 首页样式污染 | CSS 必须限定在练习页作用域 |
| 拼音未完整就报红 | 拼音判断必须区分完整正确、部分正确、错误 |
| TTS 批量返工 | 先离线检查，再 probe，再批量 |
| GitHub 固化过早 | 必须通过本地总验收闸门 |
