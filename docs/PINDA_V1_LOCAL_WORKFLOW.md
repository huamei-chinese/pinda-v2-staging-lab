# Pinda V1 本地执行流程规范

所有 Pinda V1 本地任务必须先读取并遵守：

```text
docs/PINDA_V1_LOCAL_ACCEPTANCE_GATE.md
```

## 1. 标准流程

1. 读取本地总验收闸门。
2. 明确本轮允许修改范围。
3. 定位真实数据源。
4. 定位唯一渲染函数或唯一解析函数。
5. 如果涉及素材，先离线检查素材。
6. 小批量本地导入。
7. 启动本地预览。
8. 运行自动测试。
9. 按人工截图清单验收。
10. 老板确认。
11. 才允许进入 GitHub 固化。

## 2. 定位真实数据源

每次修改 UI 或素材绑定前，必须确认：

1. 页面当前读取哪个文件。
2. 字段来自 manifest、lesson 文件、loader，还是旧内置数据。
3. 是否存在旧 fallback。
4. 测试是否覆盖真实页面路径。

禁止只根据新测试包判断真实页面。

## 3. 唯一函数原则

核心逻辑必须收敛到唯一出口：

1. 学习模块分类：唯一 bucket 函数。
2. 音频路径解析：唯一 audio resolver。
3. sentence segment 标签：唯一 `displaySegmentLabel(segment)`。
4. 拼音判断：唯一 pinyin validation 函数。
5. 素材路径读取：唯一 manifest / loader 入口。

禁止多个地方各自判断同一规则。

## 4. 素材本地导入流程

1. 只读验收素材包。
2. 确认 manifest、audio、README、validation_report 存在。
3. 确认 `word` / `sentence` 数量正确。
4. 确认没有 `phrase` / `world`。
5. 确认 `word` 全部为 1～4 个汉字。
6. 确认 `word` 不是全部 4 字固定表达。
7. 确认第一批测试素材至少包含 3 条 2 字核心词。
8. 确认没有为了 TTS 稳定性把 `word` 全部做成 4 字。
9. 确认每条 sentence 有 segments。
10. 确认 sentence segments 是自然拆解后的多个词块，不是整句单 segment。
11. 确认 5 个及以上汉字的 sentence 不存在 `segments.length === 1` 且 segment 等于整句的情况。
12. 确认 segments 拼接后与原句一致。
13. 确认浏览器中长句下划线数量与 segments 数量一致。
14. 确认 normal / slow 音频存在且非 0KB。
15. 验收通过后，才复制到本地测试目录。
16. 未通过本地预览，禁止 GitHub 固化。

## 5. 前端修复流程

1. 先复现问题。
2. 定位真实渲染代码。
3. 找到字段来源。
4. 检查旧 fallback。
5. 增加或更新回归测试。
6. 做最小修复。
7. 运行测试。
8. 本地浏览器人工预览。
9. 老板确认。

## 6. 人工截图问题处理流程

出现截图问题后，必须记录：

1. 截图问题文字。
2. 所在页面。
3. 真实数据源。
4. 渲染函数。
5. 修复函数。
6. 回归测试用例。

示例必须固化：

1. “发给我”不能只显示“动词”，必须显示“短语 · 动词”。
2. “这个岗位”不能显示“生词”，必须显示“短语 · 名词”。
3. “工作经验”不能显示“生词”，必须显示“短语 · 名词”。
4. 首页 hero 标题不能被下划线污染。
5. 拼音正确但不完整不能显示红色。
6. 长句不能只有一条整句下划线，必须按 `segments` 显示多个独立输入块。
7. 如果素材音频正确但 `segments` 错误，只重建 manifest / 交接文档 / validation_report，不重新跑 TTS。

## 7. GitHub 固化流程

只有满足以下条件才可进入 GitHub 固化：

1. 自动测试通过。
2. 本地预览通过。
3. 老板人工确认通过。
4. 安全文件未进入 Git。
5. 修改范围清楚。
6. 本地临时素材和正式提交内容已分层。

未经老板确认，禁止 commit / push / PR / merge。
